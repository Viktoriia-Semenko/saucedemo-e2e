# Тест-кейси основних user-flow - saucedemo.com

Об'єкт тестування: https://www.saucedemo.com/
Тестові дані: `standard_user` / `secret_sauce` (і додаткові користувачі нижче)

---

## TC-01: Наскрізний сценарій покупки (happy path)

| Поле | Значення |
|------|----------|
| **ID** | TC-01 |
| **Назва** | Успішне оформлення замовлення авторизованим користувачем |
| **Пріоритет** | Високий: критичний бізнес флоу |
| **Тип** | Позитивний, наскрізний (E2E) |
| **Передумови** | Користувач не авторизований, відкрита сторінка логіну |

**Кроки:**

1. Ввести логін `standard_user` та пароль `secret_sauce`, натиснути **Login**.
2. Додати товар "Sauce Labs Backpack" кнопкою **Add to cart**.
3. Відкрити кошик (іконка кошика у шапці).
4. Натиснути **Checkout**, заповнити дані: ім'я `Test`, прізвище `User`, індекс `01001`, натиснути **Continue**.
5. На кроці огляду натиснути **Finish**.

**Очікуваний результат:**

- Після логіну відкривається сторінка `/inventory.html`.
- Бейдж кошика показує `1`.
- У кошику одна позиція - «Sauce Labs Backpack».
- Після Finish відкривається `/checkout-complete.html` з повідомленням **«Thank you for your order!»**.

## TC-02: Авторизація користувача (позитив + негатив)

| Поле | Значення |
|------|----------|
| **ID** | TC-02 |
| **Назва** | Перевірка входу з валідними та невалідними даними |
| **Пріоритет** | Високий |
| **Тип** | Позитивний + негативний (data-driven) |
| **Передумови** | Відкрита сторінка логіну |

**Сценарії:**

|  | Логін | Пароль | Очікуваний результат |
|---|-------|--------|----------------------|
| 2.1 | `locked_out_user` | `secret_sauce` | Помилка: *Epic sadface: Sorry, this user has been locked out.* Залишаємось на сторінці логіну |
| 2.2 | `standard_user` | `wrong_password` | Помилка: *Epic sadface: Username and password do not match any user in this service* |
| 2.3 | *(пусто)* | *(пусто)* | Помилка: *Epic sadface: Username is required* |
| 2.4 | `standard_user` | `secret_sauce` | Успішний вхід, відкривається `/inventory.html` |

**Очікуваний результат (загальний):** невалідні дані блокують вхід і показують коректне повідомлення про помилку; валідні - відкривають каталог товарів

---

## Чому саме ці два кейси??

- **TC-01** покриває головний бізнес-сценарій (авторизація → каталог → кошик → checkout → підтвердження) - це end-to-end потік через 4 сторінки
- **TC-02** покриває вхідну точку застосунку та обробку помилок; побудований як data-driven, щоб одним тестом перевірити кілька наборів даних
